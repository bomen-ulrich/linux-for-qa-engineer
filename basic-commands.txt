To check where (in which directory) you are currently working:
>>$pwd
pwd == stand for present working directory

Once you know what is the active directory at this moment, you want to work in an organize way.
First check what are all the availables directories: use the command:
>>$ls -al

To stay organize, we want to move to the "Documents" directory to start working:
>>$cd Documents
And we verify that we have successfully move
>>$pwd

From there, we create a new directory for our project or work (to stay clean):
>>$mkdir qa-engineer-linx
Then we move into that directory:
>>$cd qa-engineer-linux

Now we create the first file name "basic-commands.txt" about basic linux commands a QA Engineer must know:
>>$touch basic-commands.txt
Now we open that file:
>>nano basic-commands.txt

Since we are following the exact intructions on this file, let's exit and start practising!!
Using the content of this file, here is what to practice on your terminal:

Phase 1: Let's get familiar with the Shell
- As a QA,  the shell is where we trigger tests, we read and see the results
- first to check which user owns a runing process,
    >>$whoami
    It give us back the current active user
    It means that all the currently running process run with that user level of permission
    This is crucial for solving permission bugs == change the user or grant more permission to the user

- Next, to check the environment variables
	>>$printenv
	It show us all the system variable with their current values
	We can use those variable in our script, bash script
	The command to check one of those variable value, we can type the command: echo $USER - echo $PATH

- Next, to verify the shell type we are working on:
	>>echo $SHELL
	This is also another use of environment variable, since (SHELL) is an env. variable
	Remark to respect the cast, if not, it won't work since SHELL is not shell

- Next, to check the command history:
	>>history
	It will show you the list of all the commands that were type using this current USER session
	It also help us to see and  reuse complex test commands without retyping them

Phase 1 task: Create a simple alias
	>>alias checklogs='tail -f /var/log/syslog'
	This command create a shortcut or a short form command name "checklogs" for the long command "tail -f /var/log/syslog
	Now instead of typing that long command, we can just type "checklogs"
	Whenever we are typing "checklogs" we are basically typing "tail -f /var/log/syslog
	We are now seeing a real-time monitoring of the system
	We can open an application and come back to the terminal and see how we are directly been informed
	the "tail" command allow us to see the log files stored inside the "syslog" directory
	To remove or cancel that alias: >>unalias checklogs

Phase 2:The filesystem hierachy or the data map
- First thing first. Everything in linux is a file. if a bug occurs, it is usually because a file is in the wrong place or has the wrong "rules"
- The "/bin" directory:
	>>ls /bin
	It show us a bunch of program or commands to execute
	bin == binary and binary == executable, program
	Here we can find program like, ls - rm - ping - mkdir - grep - cp ...

- The "/etc" directory
	>>ls /etc
	It is where all the app/programs configurations files live
	If an app won't start, check here
	If we want to change the configuration of a programm, come here
	It is also very important for QA Engineer: for setting up, modifiying and validating environments
	QA Engineer use it to configure services, manage database connections and control test application behavior
	>>/etc/default == contains default settings for daemons and services ==useful for validating default configuration
	>>/etc/hosts ==used to map IP addresses to hostnames for testing microservices 
	>>/etc/apt/sources.list.d ==use to configure software repositories for automated installation of test dependencies
	>>/etc/network ==essential for testing network configurations and interface changes
	Some common QA Automation tasks with /etc:
		>>For env setup:Use Ansible to automate changes within /etc files for consistent test environments
		>>For validation : by running sripts to verify that specific security or configuration is ok or correctly applied
		>>For Dependency Management: by ensuring the correct versions of testing tools are installed
	

- The "/var/log" directory
	>>ls /var/log
	It is a very important directory for QA Engineer
	It helps us understand what happened on the server,or on the CI environment when a test fails
	It helps us confirm the origin of the problem/fail: whether it is due to the App, the API, the permissions, the network, the Os, the deployment setup.
	A QA checks /var/log for system messages, authentication events, database log, pakages/install issues, web server logs
	If you test backend systems, APIs, microservices,or deployment pipeline
	QA engineer should know how to use the below commands to inspect logs quickly
		>>tail -f
		>>grep
		>>less
	For example: if a test say "login failed", checking the logs can tell us whether the issue is a bad password, an expired token, a database error, or an SSH authentication problem


- The "/tmp" directory
	The running apps store their temporary data here
	It acts as a sandbox for temporary storage preventing clutter in productive directories 
	If this directory fill up due to space issue, apps will crash



Phase 3: The Kernel & Processes 
- The " ps aux" : list process
	It is a snapshot of evry living program
	It helps us to find the PID = Process ID to locate it log in /var/log or just to terminate/kill that process
	It helps us find if a backend(webserver, database) process is actually running
	It helps us to pinpoint ressources that are using too much %CPU or $MEM
		>>ps aux | grep "my-app-name"		>>to find if my-app-name is runing
		>>ps aux --sort=-%mem | head 11		>>to find the top 10 memomy consuming processes
		>>ps aux | awk '$8=="z" {print}		>>to identifie zombies process = there are processes that have finished but are still taking up space in the process table



- The "top or htop" : Live monitor
	It give us realtime hearbeat of the system
	It show us which test is eating all your RAM
	htop is the coloured version of top, more visual
	It is a real time monitor to bridge the gar between what is happening on the screen and what is hapeniing inside the server


- The "dmes" : the kernel logs
	It shows kernel-level event (hardware, drive, memory, USB, Iot devices)
		>>sodu dmesg -l err,warn  ===filter the output to only show the errors and warnings


- The "kill" : kill signal
	It force-close a "zombie process that refuses to stop after a failed test
		>>kill <PID>  : the polite request to ask the app to shut down giving it time to save it works, to cloase it database connections, to delete temporary files in /tmp
		>>kill -9 <PID> :the immediate action, the kernel kills the process instantly
	Checklist process termination:
		>>kill <PID>
		>>ps aux | grep <PID>  check the process is gone
		>>kill -9 <PID>  if the process still exist
		>>dmesg verify the logs
